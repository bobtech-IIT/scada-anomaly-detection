# SCADA Anomaly Detection вЂ” Full-Stack Web Application Plan

## Overview

Transform the existing Python CLI-based SCADA anomaly detection proof-of-concept into a **production-ready full-stack web application** with:

- **Frontend:** React + TypeScript + Vite + TailwindCSS (dashboard, charts, reports)
- **Backend:** FastAPI (Python) wrapping the existing ML pipeline
- **Database:** PostgreSQL via SQLAlchemy ORM with Alembic migrations
- **Infrastructure:** Docker + Docker Compose for one-command deployment

---

## Repository Context

The existing code in `scada-anomaly-detection/` provides:
- `src/generate_scada_data.py` вЂ” generates 5,000-row synthetic wind turbine telemetry CSV
- `src/anomaly_model.py` вЂ” Isolation Forest anomaly detector (scikit-learn)
- `src/alert_system.py` вЂ” template-based incident report generator
- `main.py` вЂ” CLI pipeline orchestrator

All existing ML logic will be **preserved and integrated directly** into the backend.

## Database Schema

### Table: `telemetry_records`
| Column          | Type        | Notes                            |
|-----------------|-------------|----------------------------------|
| id              | SERIAL PK   |                                  |
| turbine_id      | VARCHAR(50) | e.g. "T-402"                     |
| timestamp       | TIMESTAMPTZ |                                  |
| rotor_rpm       | FLOAT       |                                  |
| gearbox_temp_c  | FLOAT       |                                  |
| vibration_mm_s  | FLOAT       |                                  |
| created_at      | TIMESTAMPTZ | auto                             |

| Column            | Type        | Notes                           |
|-------------------|-------------|---------------------------------|

| detection_run_id  | UUID        | groups records from same run    |
| detected_at       | TIMESTAMPTZ | auto                            |

### Table: `incident_reports`
| Column        | Type        | Notes                             |
|---------------|-------------|-----------------------------------|
| id            | SERIAL PK   |                                   |
| detection_id  | INT FK      | в†’ anomaly_detections.id           |
| turbine_id    | VARCHAR(50) |                                   |
| report_text   | TEXT        | generated report body             |
| severity      | VARCHAR(20) | "HIGH" / "MEDIUM" / "LOW"         |
| created_at    | TIMESTAMPTZ | auto                              |

### Table: `model_runs`
| Column          | Type        | Notes                           |
|-----------------|-------------|---------------------------------|

## Backend API Endpoints

### Dashboard
- `GET /api/dashboard/stats` вЂ” total records, anomaly count, open reports, model status

### Telemetry
- `GET /api/telemetry` вЂ” paginated list (filters: turbine_id, date range, anomaly flag)
- `GET /api/telemetry/{id}` вЂ” single record
- `POST /api/telemetry/upload` вЂ” upload CSV file (multipart/form-data)
- `POST /api/telemetry/seed` вЂ” generate & seed synthetic data into DB
- `GET /api/telemetry/chart` вЂ” time-series data for charts (sampled/aggregated)

### Anomaly Detection
- `POST /api/detect` вЂ” run detection on all un-processed records, returns detection run UUID
- `GET /api/detect/runs` вЂ” list detection runs with summary counts
- `GET /api/detect/runs/{run_id}` вЂ” paginated anomaly results for a run

### Incident Reports
- `GET /api/reports` вЂ” paginated list, filterable by severity/turbine_id
- `GET /api/reports/{id}` вЂ” full report text

### Model
- `POST /api/model/train` вЂ” retrain Isolation Forest (body: `{ contamination: 0.02 }`)
- `GET /api/model/status` вЂ” latest model_run record

### Pages
1. **Dashboard** (`/`) вЂ” KPI cards (total records, anomalies, open reports) + time-series chart + recent anomalies table

4. **Incident Reports** (`/reports`) вЂ” list with severity badges; click to read full report

- `AnomalyTable` вЂ” sortable table with row highlighting for anomalies
- `KPICard` вЂ” stat summary card
- `ReportCard` вЂ” incident report card with severity badge
- `UploadZone` вЂ” drag-and-drop CSV uploader
- `Navbar` / `Sidebar` вЂ” navigation

## Key Design Decisions

- **Async SQLAlchemy** used throughout backend for production-level concurrency
- **Detection runs** are identified by UUID so the UI can poll or refresh results per run
- **Severity** is derived from anomaly values: HIGH if gearbox_temp > 90В°C or vibration > 4 mm/s, MEDIUM if temp > 80В°C or vibration > 3 mm/s, else LOW
- **CSV upload** parses the same columns as `generate_scada_data.py` so any compatible file works out-of-the-box
- **Model persistence** uses `joblib` to `backend/models/` volume-mounted in Docker
- All environment config via `.env` (DATABASE_URL, SECRET_KEY, etc.)
